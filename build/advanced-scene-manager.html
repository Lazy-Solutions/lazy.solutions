How does Advanced Scene Manager 3 change scene management?

ASM changes the level at which developers think about scenes. Instead of treating scenes as isolated files, it makes scene relationships explicit.

Before ASM: load this scene, wait for this callback, keep this object alive, call this manager, remember to unload it later. After ASM: this collection groups these scenes, this profile defines startup, this scene persists through this lifecycle, and this transition follows a structure.

The simplest way to understand ASM3

Scenes stop being individual files and become connected parts of a system. A project is not just a list of scenes; it is a collection of relationships between scenes.

That is the real shift: ASM3 provides the place where scene decisions live, instead of scattering them across utilities and scripts.

What does ASM actually replace in your workflow?

ASM is not just a better loader. It replaces scattered scene control scripts with one organized system for scene flow.

A loading screen tool is a utility. A transition helper is a utility. ASM3 is the system where those scene decisions belong.

How do I load a scene in Unity with ASM3?

ASM3 makes scene loading part of the scene relationship system, instead of leaving raw SceneManager calls scattered across code. For example:

public class SceneLoader : MonoBehaviour
{
    [SerializeField] private SceneCollection gameplayCollection;
    [SerializeField] private Scene mainMenuScene;

    public void OpenGameplay()
    {
        gameplayCollection.Open();
    }

    public void OpenMainMenu()
    {
        mainMenuScene.Open();
    }
}

Read the ASM documentation for scene collection and scene open workflows.

This is the core Unity load scene pattern for ASM3: use scene collections and scene open methods, not manual SceneManager plumbing.

These calls tell ASM3 what to open, while the system handles transition order, persistence, cleanup, and the rest of the scene flow.
